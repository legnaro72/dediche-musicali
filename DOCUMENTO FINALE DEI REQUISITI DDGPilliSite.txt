DOCUMENTO FINALE DEI REQUISITI
Sito gratuito per dediche musicali automatiche
Architettura: GitHub Pages + GitHub Actions + Astro + Python + Google Sheet
________________


1. Scopo del progetto
Realizzare un sito web pubblico dedicato alla pubblicazione di dediche musicali giornaliere.
Il sito deve essere:
* completamente gratuito da mantenere;
* pubblicato su GitHub Pages;
* aggiornato automaticamente ogni giorno;
* graficamente moderno, accattivante e adatto a contenuti musicali/emozionali;
* gestibile senza dover modificare ogni giorno il codice;
* integrato con link audio esterni, per esempio Spotify, YouTube Music, SoundCloud o cloud;
* integrato con un pulsante “Votami” collegato a un Google Form;
* dotato di archivio dediche;
* dotato di sistema di backup;
* predisposto per una futura interfaccia manuale tramite Streamlit.
L’obiettivo principale è che l’utente possa preparare in anticipo settimane o mesi di dediche e lasciare che il sito pubblichi automaticamente la dedica corretta ogni giorno.
________________


2. Vincoli obbligatori
2.1 Costo
La soluzione deve avere costo operativo pari a:
0 euro/mese
Non devono essere richiesti:
* hosting a pagamento;
* VPS;
* database a pagamento;
* WordPress a pagamento;
* servizi cloud a pagamento;
* backend server persistente.
________________


2.2 Account da usare
La soluzione deve essere realizzata usando un account GitHub già esistente.
Il sito pubblico dovrà essere disponibile con URL del tipo:
https://USERNAME.github.io/NOME-REPOSITORY/
Il dominio personalizzato è opzionale e non deve essere necessario per il funzionamento del progetto.

6. Struttura del Google Sheet
Il Google Sheet deve contenere una riga per ogni dedica.
Ogni riga rappresenta una dedica pubblicabile in una data specifica.
6.1 Colonne obbligatorie
Il Google Sheet deve contenere queste colonne:
id
date
status
song_title
artist
dedication_title
dedication_text
audio_url
audio_type
vote_url
image_mode
image_source
short_phrase
tags
seo_title
seo_description
image_alt
________________


6.2 Descrizione dei campi
id
Identificativo univoco della dedica.
Formato consigliato:
2026-05-12-nome-canzone
Regole:
* obbligatorio;
* univoco;
* senza spazi;
* usare solo lettere minuscole, numeri e trattini.
________________


date
Data di pubblicazione della dedica.
Formato obbligatorio:
YYYY-MM-DD
Esempio:
2026-05-12
La data deve essere interpretata secondo il fuso orario:
Europe/Rome
________________


status
Stato della dedica.
Valori ammessi:
draft
scheduled
published
disabled
Significato:
draft      = bozza, non pubblicare
scheduled  = pubblicabile automaticamente quando arriva la data
published  = già pubblicata
disabled   = esclusa dalla pubblicazione
________________


song_title
Titolo della canzone.
Campo obbligatorio.
________________


artist
Nome dell’artista.
Campo obbligatorio.
________________


dedication_title
Titolo della dedica.
Esempio:
La dedica del giorno
Campo obbligatorio.
________________


dedication_text
Testo completo della dedica.
Campo obbligatorio.
Deve supportare:
* testo multilinea;
* emoji;
* accenti;
* apostrofi;
* caratteri italiani.
________________


audio_url
Link alla traccia audio.
Campo obbligatorio.
Può essere:
* link Spotify;
* link YouTube Music;
* link SoundCloud;
* link MP3 pubblico;
* link a file cloud;
* altro link audio pubblico.
Il sito non deve caricare brani musicali protetti dentro il repository. Deve usare link o embed esterni.
________________


audio_type
Tipo di sorgente audio.
Valori ammessi:
spotify
youtube
soundcloud
mp3
cloud
other
________________


vote_url
Link al Google Form per votare.
Campo facoltativo solo se è configurato un link Google Form predefinito.
Regola:
Ogni dedica pubblicata deve avere sempre un pulsante “Votami”.
Se vote_url è vuoto, il sistema deve usare il valore predefinito configurato nei GitHub Secrets.
________________


image_mode
Modalità immagine.
Valori ammessi:
auto
upload
none
Significato:
auto    = immagine generata automaticamente da Python
upload  = immagine fornita manualmente
none    = usare immagine placeholder
________________


image_source
Percorso o URL dell’immagine caricata manualmente.
Obbligatorio solo se:
image_mode = upload
________________


short_phrase
Frase breve da inserire nell’immagine generata.
Esempio:
Una canzone per ricordare ciò che conta.
Campo facoltativo.
________________


tags
Lista tag separata da virgole.
Esempio:
amore,ricordi,estate
________________


seo_title
Titolo SEO della pagina.
Se vuoto, deve essere generato automaticamente usando:
song_title + artist + date
________________


seo_description
Descrizione SEO della pagina.
Se vuota, deve essere generata automaticamente usando una sintesi della dedica.
________________


image_alt
Testo alternativo dell’immagine.
Obbligatorio per accessibilità.
Se vuoto, deve essere generato automaticamente usando titolo canzone, artista e data.
________________


7. Validazione dati
Prima di pubblicare una dedica, lo script Python deve validare i dati.
7.1 Regole obbligatorie
La pubblicazione deve fallire se:
* id è vuoto;
* id è duplicato;
* date è vuota;
* date non è nel formato YYYY-MM-DD;
* esistono due dediche attive per la stessa data;
* song_title è vuoto;
* artist è vuoto;
* dedication_title è vuoto;
* dedication_text è vuoto;
* audio_url è vuoto;
* audio_url non è un URL valido;
* vote_url è vuoto e non esiste un Google Form predefinito;
* status contiene un valore non previsto;
* image_mode contiene un valore non previsto.
________________


7.2 Comportamento in caso di errore
Se la validazione fallisce:
* il sito già online non deve essere modificato;
* il deploy non deve avvenire;
* il workflow deve terminare con errore;
* l’errore deve essere leggibile nei log di GitHub Actions;
* il Google Sheet deve essere aggiornato con messaggio di errore, se possibile.
________________


8. File JSON generati
Per ogni dedica valida, il sistema deve generare un file JSON.
Percorso richiesto:
/data/dedications/YYYY-MM-DD.json
Esempio:
/data/dedications/2026-05-12.json
________________

Ogni JSON deve avere questa struttura:
{
 "id": "2026-05-12-nome-canzone",
 "date": "2026-05-12",
 "day_name": "martedì",
 "status": "published",
 "song_title": "Titolo canzone",
 "artist": "Nome artista",
 "dedication_title": "Titolo dedica",
 "dedication_text": "Testo completo della dedica",
 "audio": {
   "url": "https://...",
   "type": "spotify"
 },
 "vote": {
   "url": "https://..."
 },
 "image": {
   "path": "/images/dedications/2026-05-12.webp",
   "alt": "Dedica musicale del 12 maggio 2026"
 },

   "title": "Titolo SEO",
   "description": "Descrizione SEO"
 },
 "created_at": "2026-05-12T06:00:00+02:00",
 "updated_at": "2026-05-12T06:00:00+02:00"

9. Regole di pubblicazione giornaliera
9.1 Fuso orario
Il sito deve usare come data logica:
Europe/Rome
La dedica del giorno deve essere scelta sulla base della data italiana, non della data UTC.
________________


9.2 Dedica pubblicabile
Una dedica è pubblicabile automaticamente se:
date = data odierna Europe/Rome
status = scheduled
________________


9.3 Pubblicazione automatica
Ogni giorno GitHub Actions deve:
1. leggere il Google Sheet;
2. trovare la dedica del giorno;
3. validare i dati;
4. generare il JSON;
5. generare o copiare l’immagine;
6. aggiornare lo stato della dedica;
7. costruire il sito Astro;
8. pubblicare il sito su GitHub Pages;
9. creare backup;
10. scrivere log chiari.
________________


9.4 Nessuna dedica trovata
Se per il giorno corrente non esiste una dedica valida:
* il sito online deve restare invariato;
* la homepage deve continuare a mostrare l’ultima dedica pubblicata;
* il workflow deve registrare un warning;
* non devono essere cancellati contenuti esistenti.
________________

22. GitHub Actions richieste
Devono essere previsti almeno tre workflow.
________________


22.1 deploy.yml
Esegue il deploy del sito quando viene fatto push su main.
Trigger:
on:
 push:
   branches:
     - main
Funzioni:
* installare dipendenze Node;
* buildare Astro;
* pubblicare su GitHub Pages.
________________


22.2 daily-publish.yml
Esegue la pubblicazione automatica giornaliera.
Trigger:
on:
 schedule:
   - cron: "10 4 * * *"
 workflow_dispatch:
Nota: il cron GitHub usa UTC. Lo script Python deve comunque calcolare la data corretta usando Europe/Rome.
Il workflow deve supportare anche esecuzione manuale con input:
date
force_republish
dry_run
________________


22.3 backup.yml
Esegue backup automatici.
Trigger:
on:
 schedule:
   - cron: "30 4 * * *"
 workflow_dispatch:
Funzioni:
* creare archivio zip;
* salvare backup come artifact GitHub Actions;
* opzionalmente creare GitHub Release di backup;
* non appesantire inutilmente il repository con zip giornalieri permanenti.
________________


23. Backup
23.1 Obiettivo
Il sistema deve permettere backup e ripristino del sito.
Il backup deve consentire di recuperare:
* dediche;
* immagini;
* codice sorgente;
* configurazioni;
* script;
* workflow;
* file pubblicabili;
* configurazione Astro.
________________


23.2 Contenuto del backup
Ogni backup deve includere:
data/
public/images/dedications/
src/
scripts/
.github/workflows/
package.json
astro.config.mjs
requirements.txt
README.md
Opzionale:
dist/
La cartella dist/ è opzionale perché il sito può essere rigenerato dal sorgente.
________________


23.3 Tipi di backup
Devono esistere tre livelli di backup.
Livello 1 — Git
Ogni modifica deve essere versionata da Git.
Questo permette rollback tramite commit precedenti.
________________


Livello 2 — Artifact GitHub Actions
Il workflow backup.yml deve creare un file:
backup-YYYY-MM-DD.zip
da scaricare dagli artifact GitHub Actions.
________________


Livello 3 — Release GitHub opzionale
Per backup più stabili, lo sviluppatore deve prevedere la possibilità di creare release di backup.
Esempio:
backup-2026-05-12
________________


23.4 Restore
Lo sviluppatore deve documentare una procedura di ripristino.
Il restore deve permettere:
* ripristino completo del sito;
* ripristino di una singola dedica;
* ripristino di una singola immagine;
* rollback a un commit precedente.
________________

26. Sicurezza

GOOGLE_SERVICE_ACCOUNT_JSON
GOOGLE_SHEET_ID
DEFAULT_VOTE_URL
SITE_NAME
SITE_URL
Il token GitHub standard può essere quello fornito automaticamente da GitHub Actions:
GITHUB_TOKEN

26.2 Google Sheet
Il Google Sheet deve essere condiviso solo con:
* proprietario;
* eventuale sviluppatore;
* service account usato dallo script.

26.3 Validazione URL

https://

34. Obiettivo finale operativo
Alla fine del progetto, l’utente deve poter lavorare così:

3. aggiunge titolo, artista, testo, link audio e link voto;
4. salva il foglio;
5. non tocca più il sito;
6. ogni giorno GitHub Actions pubblica automaticamente la dedica corretta;
7. il sito resta pubblico su GitHub Pages;
8. in caso di errore, il sito precedente resta online;
9. in caso di problemi, è possibile ripristinare un backup;
10. in futuro, la gestione potrà essere resa più comoda tramite app Streamlit.
